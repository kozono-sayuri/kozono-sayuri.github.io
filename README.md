# 音樂網站更新規則

## 任務目的

依照既有 HTML 範本，新增或更新 Co-workersword 曲目目錄及音樂創作說明頁。

- Co-workersword 目錄頁：<https://kozono-sayuri.github.io/music/cb-workersword/>
- 音樂創作說明頁範例：<https://kozono-sayuri.github.io/music/cb-workersword/005.html>

## 正式範本

- Co-workersword 目錄頁：`templates/co-workersword-directory-template.html`
- Co-workersword 音樂創作說明頁：`templates/co-workersword-song-page-template.html`
- 頁面文字原稿：`docs/content.md`
- 更新需求與操作說明：`docs/update-notes.md`

## 目錄頁更新規則

1. 頁首標題、副標題、導覽列、Story Concept、Tracks & Notes 說明及頁尾皆為固定內容。
2. 往後只更新或新增曲目介紹卡，不得自行修改其他固定內容。
3. 曲目介紹卡一律採倒序排列，最新或編號最大的卡片放在最前面。
4. 新增卡片時，應插入目前最新卡片之前。
5. 每張卡片必須包含曲目編號、歌名、簡介、詳細頁連結及進入提示文字。

## 音樂創作說明頁更新規則

### 既有歌曲資料保護（禁止覆蓋）

1. 每一個歌曲編號都是獨立且永久的資料頁；例如更新 `007` 時，只能建立或修改 `007.html`，不得以新歌曲內容覆寫 `006.html`、`005.html` 或其他既有歌曲頁。
2. 「更新編號 XXX」的意思，是更新該編號所對應的 `XXX.html`；除非使用者明確指定要修正另一個既有編號，否則不得修改其他歌曲頁的歌名、正文、日期、YouTube、Suno、Facebook 或版本資料。
3. 新增歌曲時，必須先確認目標編號是否已有正式內容：
   - 若檔案不存在或只是尚未發布的空白範本，才可建立或填入新歌曲內容。
   - 若檔案已有其他歌曲的正式內容，必須停止修改並向使用者確認，不得直接取代。
4. 新頁面必須由正式範本複製後另存為對應編號檔案；不得把前一首歌曲頁另存後遺留或覆蓋其原始資料。
5. 更新目錄頁時，只能新增新歌曲卡片並依倒序重新排列；既有歌曲卡片必須完整保留，不得因加入新卡片而刪除、改寫或被新歌曲資料取代。
6. 目錄排序只改變卡片顯示位置，不改變歌曲編號、詳細頁檔名或既有頁面內容。例如加入 `007` 後，排序應為 `007 → 006 → 005`，而 `006.html` 仍永久對應原本的 006 歌曲。
7. 修改完成後，必須比對目標頁與前一編號頁，確認兩者為不同檔案、不同歌曲內容，且所有既有頁面仍可正常開啟。

### 開始前必問

每次收到「更新或新增 Co-workersword 音樂創作說明頁」的需求時，開始修改前都必須先向使用者詢問並確認以下五項資料：

1. 歌曲完成日
2. MV 完成日
3. 第一名連結
4. 第二名連結及版本編號
5. Facebook 連結

即使其他歌曲資料已齊全，仍須先完成這五項確認；不得自行猜測、沿用舊歌曲資料或略過詢問。

### 頁面規則

1. 從 `templates/co-workersword-song-page-template.html` 複製新頁面，不得直接覆寫範本。
2. 保留既有版型、CSS class、主要導覽、頁內導覽、四個內容區塊及頁尾。
3. 將所有方括號標記的欄位，例如 `[歌曲編號]`、`[中文歌名]`，替換成正式內容。
4. 四個固定內容區塊為 Overview、Lyrics & Meaning、Production Notes 及 Suno Elements。
5. 頁內導覽連結必須與 `overview`、`lyrics`、`production`、`suno` 等區塊 ID 相符。
6. 第一名連結用於頁面中的 YouTube 影片播放器，必須轉換為 `https://www.youtube.com/embed/影片ID` 格式後填入 iframe。
7. 第二名連結及版本編號放在 Lyrics & Meaning 區塊底部。
8. Production Notes 區塊底部不放第一名或第二名連結。
9. Facebook 連結放在 Suno Elements 區塊底部。
10. 第二名及 Facebook 外部連結必須確認有效，並保留 `target="_blank"` 與 `rel="noopener noreferrer"`。
11. 缺少歌曲資料時應列出缺項，不得自行捏造。

## 素材與檔案

- 圖片：`assets/images/`
- 音訊：`assets/audio/`
- 圖示：`assets/icons/`
- 字型：`assets/fonts/`

新歌曲的實際存放位置應以 GitHub 網站既有路徑為準；建立前先確認目前專案結構，不得自行假設資料夾層級。

## 共通執行規則

1. 不得自行改動全站共用版型。
2. 中文與英文內容都必須完整。
3. 缺少資料時列出缺項，不得自行捏造。
4. 修改完成後必須預覽頁面，並檢查桌面版、手機版及所有連結。
5. 未經明確授權，不得 commit、push、部署或公開發布。
6. 所有新增與更新都必須採「保留歷史資料」原則；新歌曲只能增加新檔案與新目錄卡片，不得覆蓋、挪用或清除過去歌曲資料。

## 完成回報

每次更新必須回報：

- 修改了哪些檔案
- 新增檔案的位置
- 是否完成桌面版與手機版預覽
- 是否存在缺漏資料或失效連結
- 是否已確認既有歌曲頁與目錄卡片均未遭覆蓋
- 是否已 commit、push 或部署
